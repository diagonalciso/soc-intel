Introduction

SOCint is a multi-service threat intelligence platform requiring careful deployment, monitoring, and maintenance. This manual covers setup, operations, scaling, and troubleshooting for administrators.

System Architecture:

  • API Server — FastAPI (port 8000), handles REST, GraphQL, SSE
  • Worker — APScheduler-based connector scheduler, alert matcher, IoC decay
  • Frontend — React 18 + Vite (port 3000), proxies /api to backend
  • PostgreSQL — Relational data (users, cases, rules, compliance)
  • OpenSearch — STIX 2.1 object store, full-text search
  • Redis — Cache, pub/sub for real-time alerts
  • RabbitMQ — Message queue for bulk indexing
  • Minio — S3-compatible file storage for exports
  • Tor SOCKS5 — Proxy for dark web connectors

Hardware Requirements:

  • Minimum: 4 vCPU, 8 GB RAM, 50 GB disk, 10 Mbps bandwidth
  • Recommended: 8+ vCPU, 16+ GB RAM, 200+ GB disk, gigabit network

Deployment & Setup

Docker Compose (Full Stack, Recommended)

Deploy all services in one command:

  1. Clone or pull socint repository
  2. Copy .env.example to .env and configure (see below)
  3. On Linux, set OpenSearch max_map_count once: 
    sudo sysctl -w vm.max_map_count=262144
  4. Build and start: 
    docker compose up -d --build
  5. Verify all services running: 
    docker compose ps
  6. View logs: 
    docker compose logs -f
  7. Access frontend: http://localhost:3000
  8. API docs: http://localhost:8000/docs
First Login: Admin account auto-created from SEED_ADMIN_PASSWORD in .env. Log in with email admin@socint.local and that password. Change immediately.

Standalone Backend (Development)

Run backend without Docker for active development:

cd backend
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate
pip install -r requirements.txt
export $(cat ../.env | xargs)   # Load environment
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload

# In another terminal, start the worker:
python -m app.workers.main

Frontend Only (Development)

cd frontend
npm install
npm run dev    # Vite on port 3000, proxies /api to http://localhost:8000

Generating .env File Securely

#!/bin/bash
# Generate strong secrets
SECRET_KEY=$(python -c "import secrets; print(secrets.token_hex(32))")
CONNECTOR_API_KEY=$(python -c "import secrets; print(secrets.token_urlsafe(32))")

cat > .env << EOF
SECRET_KEY=$SECRET_KEY
CONNECTOR_API_KEY=$CONNECTOR_API_KEY
SEED_ADMIN_PASSWORD=$SEED_ADMIN_PASSWORD

TOR_PROXY=socks5://tor:9050
OTX_API_KEY=
NVD_API_KEY=
VIRUSTOTAL_API_KEY=
EOF

chmod 600 .env

Database & Storage

PostgreSQL Setup

PostgreSQL stores metadata: users, cases, rules, compliance tags, audit logs.

Initial Schema Setup:
  1. Database migrations run automatically on FastAPI startup via Alembic
  2. First run creates all tables
  3. To verify: 
    psql -h localhost -U socint -d socint -c "\dt"

Key Tables:

  • users — User accounts, roles, organizations
  • cases — Investigation cases with metadata
  • case_tasks — Tasks within cases
  • case_comments — Case discussion/notes
  • case_observables — Artifacts in cases (IOCs, etc.)
  • detection_rules — Sigma, YARA, custom rules
  • alert_rules — Rules for triggering alerts on new IOCs
  • compliance_frameworks — NIST 800-53, CSF tags
  • api_keys — User API keys (hashed)

PostgreSQL Backup & Restore

Daily Backup: 
docker compose exec -T postgres pg_dump -U socint -d socint > socint_backup_$(date +%Y%m%d).sql
Restore from Backup: 
docker compose exec -T postgres psql -U socint -d socint < socint_backup_20240115.sql

Or use automated backups: Set up pg_dump in cron or use AWS RDS automated snapshots.

OpenSearch Setup

OpenSearch stores STIX 2.1 objects (IOCs, malware, threat actors, etc.) with full-text search.

Index Management:
  1. Indices created automatically on startup
  2. Index policy: stix-objects-* (time-based rollover daily)
  3. View indices: 
    curl -s http://localhost:9200/_cat/indices | grep stix
  4. Check health: 
    curl -s http://localhost:9200/_cluster/health

Tuning for Large Datasets:

  • Shard count: Set to number of data nodes (or 1 for single node)
  • Replica count: Set to 1 for redundancy, 0 for dev
  • Refresh interval: Increase to 30s for better indexing throughput on bulk imports
curl -X PUT http://localhost:9200/stix-objects-2024-01 -H "Content-Type: application/json" -d '{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1,
    "refresh_interval": "30s"
  }
}'

OpenSearch Backup & Restore

Create Snapshot Repository: 
curl -X PUT http://localhost:9200/_snapshot/backup -H "Content-Type: application/json" -d '{
  "type": "fs",
  "settings": { "location": "/snapshots" }
}'
Snapshot All Indices: 
curl -X PUT http://localhost:9200/_snapshot/backup/snapshot_$(date +%Y%m%d)
Restore Snapshot: 
curl -X POST http://localhost:9200/_snapshot/backup/snapshot_20240115/_restore

Data Retention Policies

Older IOCs consume storage. Implement retention:

Data Type Retention Rationale
Fresh IOCs (<30 days) Keep all High relevance
Aged IOCs (30–90 days) Keep with reduced confidence Decay score applied daily
Very old IOCs (>90 days) Archive or delete if no sightings Low relevance, save storage
Case closed >1 year Archive to cold storage Compliance (keep 7 years if regulated)

  1. Click "Settings" (admin only) → "Users"
  2. Click "+ Invite User"
  3. Enter email, username, select role
  4. System sends signup link (user completes registration)

Via API:

  -H "Content-Type: application/json" \
  -d '{
    "email": "analyst@example.com",

  }'

Disabling Users

Deactivate without deletion (preserves audit trail):

  1. Settings → Users → click user → toggle "Active"
  2. User cannot login; cases/history remain

Organizations & Team Isolation

If multi-tenant, assign users to organizations:

  1. Create organization in admin panel
  2. Invite users to organization
  3. Users only see intel/cases within their org (unless shared)
  4. Note: This is a roadmap feature; current version is single-org

Security & Authentication

JWT Token Management

SOCint uses JWT (JSON Web Tokens) for stateless authentication:

  • Token Lifetime: 24 hours by default (configurable in config.py)
  • Refresh Tokens: Support for sliding session (token renewed on each request)
  • Signing Algorithm: HS256 (HMAC with SHA-256)
  • Rotation: Change SECRET_KEY in .env to invalidate all tokens (forces re-login)

API Key Security

API keys are hashed and stored securely:

  • Generation: 32 random bytes, base64-encoded
  • Storage: Only hash stored in DB, raw key shown once at creation
  • Rotation: Users can revoke and create new keys anytime
  • Scoping: Keys inherit user's permissions; no separate scopes yet (roadmap)

Revoking Compromised Keys:

curl -X DELETE http://localhost:8000/api/auth/api-keys/{key_id} \
  -H "Authorization: Bearer USER_TOKEN"

Password Policy

Implement strong password requirements:

  • Minimum Length: 12 characters
  • Complexity: Uppercase, lowercase, number, special character
  • Expiration: 90 days (optional, can disable)
  • History: Don't allow reuse of last 5 passwords

Enforce in config.py; bcrypt hashing with salt for storage.

TLS/SSL Setup

For production, require HTTPS:

version: '3'
services:
  api:
    environment:
      - HTTPS_ENABLED=true
      - CERT_PATH=/certs/server.crt
      - KEY_PATH=/certs/server.key
    volumes:
      - /path/to/certs:/certs:ro
    ports:
      - "8443:8443"  # HTTPS port

  frontend:
    environment:
      - API_URL=https://api:8443/api
    ports:
      - "443:443"

Or use reverse proxy (nginx/Caddy) for TLS termination.

Rate Limiting & DDoS Protection

Enable rate limiting on sensitive endpoints:

  • Login: 5 attempts per minute per IP
  • API calls: 1000 requests/min per user (default)
  • File uploads: 100 MB max file size

Configure in config.py or behind nginx reverse proxy.

Performance & Scaling

PostgreSQL Optimization

For large case/rule datasets, tune PostgreSQL:

Parameter Small (<100K records) Large (>1M records)
shared_buffers 256 MB 4–8 GB (25% of RAM)
effective_cache_size 1 GB 16 GB (50% of RAM)
work_mem 4 MB 64–256 MB
maintenance_work_mem 64 MB 1–2 GB
wal_buffers 16 MB 64–256 MB

Apply in docker-compose.yml:

postgres:
  image: postgres:15
  command:
    - "-c"
    - "shared_buffers=4GB"
    - "-c"
    - "effective_cache_size=16GB"
    - "-c"
    - "work_mem=128MB"

OpenSearch Tuning

For billions of STIX objects:

  • Heap Size: Set to 50% of available RAM (max 31 GB), e.g., -Xmx16g -Xms16g
  • Shard Count: 1 shard per 40 GB of data (~25 million STIX objects)
  • Refresh Interval: Increase from 1s to 30s for bulk indexing
  • Merge Settings: Tune segment merging to reduce CPU during bulk ops
opensearch:
  image: opensearchproject/opensearch:2.0.0
  environment:
    - discovery.type=single-node
    - OPENSEARCH_JAVA_OPTS=-Xmx16g -Xms16g
    - plugins.security.disabled=true  # Use external auth instead

Redis Cache Optimization

Redis caches API responses, enrichment results, and session data:

  • Memory Policy: Set maxmemory-policy=allkeys-lru (evict least-used keys when full)
  • Persistence: Enable RDB snapshots for recovery: save 300 10 (snapshot if 10 writes in 300s)
  • Monitoring: Watch memory usage; scale up if hitting limit

Configure in docker-compose.yml:

redis:
  command: redis-server --maxmemory 4gb --maxmemory-policy allkeys-lru

Horizontal Scaling (Multiple Workers)

Run multiple worker instances for parallel connector execution:

version: '3'
services:
  worker-1:
    build: ./backend

Troubleshooting

API Won't Start

Symptom: docker compose logs api shows error

Common Causes:

  • Database unreachable: Check DATABASE_URL, verify postgres running
  • OpenSearch unreachable: Verify opensearch service, check firewall
  • Missing .env: Copy .env.example → .env and configure
  • Port already in use: Kill process: lsof -i :8000

Solution:

docker compose down
docker compose up --build

Connectors Not Running

Symptom: No new IOCs ingested

Debug Steps:

  1. Check worker is running: docker compose ps | grep worker
  2. View worker logs: docker compose logs worker | grep -i error
  3. Check if connector enabled: Admin → Connectors → toggle "Enabled"
  4. Verify schedule: 0 */6 * * * means every 6 hours (adjust if needed)
  5. Run connector manually: 
    docker compose exec worker python -c "
from app.connectors.builtin.otx import OTXConnector
import asyncio
asyncio.run(OTXConnector().run())
"

Search Returns No Results

Symptom: Intel searches empty despite connectors running

Debug Steps:

  1. Check OpenSearch indices exist: 
    curl -s http://localhost:9200/_cat/indices | grep stix
  2. If no indices, connector ingestion failed. Check worker logs.
  3. Count docs in index: 
    curl -s http://localhost:9200/stix-objects-*/_count
  4. If 0 documents, manually trigger connector (see above)
  5. Verify search filter TLP level (may be hiding results with TLP:RED if user lacks access)

Database Connection Errors

Symptom: psycopg2.OperationalError: server closed connection unexpectedly

Solution:

  1. Restart postgres: docker compose restart postgres
  2. Check disk space: df -h (postgres may be full)
  3. Check PostgreSQL logs: docker compose logs postgres
  4. Clear old connections: 
    docker compose exec postgres psql -U socint -d socint -c "SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state = 'idle' AND query_start < now() - INTERVAL '1 hour';"

OpenSearch Out of Memory

Symptom: java.lang.OutOfMemoryError in OpenSearch logs

Solution:

  1. Increase heap in docker-compose.yml: OPENSEARCH_JAVA_OPTS=-Xmx32g -Xms32g
  2. Delete old indices if not needed: 
    curl -X DELETE http://localhost:9200/stix-objects-2023-*
  3. Enable index lifecycle management (ILM) to auto-archive old indices
  4. Increase system memory or scale horizontally (add nodes)

Port Conflicts

Symptom: Bind for 0.0.0.0:8000 failed: port is already allocated

Solution:

# Find and kill process using port
lsof -i :8000
kill -9 

# Or change port in docker-compose.yml
ports:
  - "8001:8000"  # Use 8001 instead

Remove deleted cases and expired API keys:

docker compose exec postgres psql -U socint -d socint << EOF
-- Delete cases closed > 2 years ago

-- Delete inactive API keys > 6 months old
DELETE FROM api_keys WHERE is_active = false AND updated_at < now() - INTERVAL '6 months';

-- Vacuum to reclaim space
VACUUM ANALYZE;
EOF

Backup & Disaster Recovery

Backup Strategy (3–2–1 Rule)

Keep 3 copies of critical data, 2 different media, 1 offsite:

Copy Location Frequency Retention
Production DB Primary server Continuous (WAL archiving) Current
Daily backup Local NAS/attached storage Daily (11 PM UTC) 30 days
Weekly backup Remote object storage (AWS S3, MinIO) Weekly (Sunday) 1 year

Automated Daily Backup Script

#!/bin/bash
# Save as /opt/socint/backup.sh

BACKUP_DIR="/mnt/backups/socint"
REMOTE_S3="s3://backup-bucket/socint"
DATE=$(date +%Y%m%d)

# Create directory
mkdir -p $BACKUP_DIR/$DATE

# Backup PostgreSQL
docker compose exec -T postgres pg_dump -U socint -d socint | gzip > $BACKUP_DIR/$DATE/postgres.sql.gz

# Backup OpenSearch (snapshot)
curl -X PUT http://localhost:9200/_snapshot/backup/snapshot_$DATE

# Upload to S3
aws s3 cp $BACKUP_DIR/$DATE s3://backup-bucket/socint/$DATE --recursive

# Cleanup old local backups (keep 30 days)
find $BACKUP_DIR -type d -mtime +30 -exec rm -rf {} \;

echo "Backup completed: $DATE"

Add to crontab:

0 23 * * * /opt/socint/backup.sh >> /var/log/socint-backup.log 2>&1

Recovery Procedures

Recover from PostgreSQL backup:

# Stop services
docker compose down

# Restore database
docker compose up -d postgres
sleep 30  # Wait for postgres to be ready
docker compose exec -T postgres psql -U socint -d postgres < socint_backup_20240115.sql

# Restart all services
docker compose up -d

Recover from OpenSearch snapshot:

# List available snapshots
curl http://localhost:9200/_snapshot/backup/_all

# Restore specific snapshot
curl -X POST http://localhost:9200/_snapshot/backup/snapshot_20240115/_restore -H "Content-Type: application/json" -d '{
  "indices": "stix-objects-*"
}'

RTO & RPO Targets

Metric Target Strategy
RTO (Recovery Time Objective) <1 hour Docker Compose re-deployment + restore from latest daily backup
RPO (Recovery Point Objective) <24 hours Daily backups at 11 PM UTC; WAL archiving for point-in-time recovery
